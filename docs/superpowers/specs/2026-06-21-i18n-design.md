# i18n Design — English + Portuguese (Brazil)

**Issue:** #13
**Date:** 2026-06-21
**Status:** Approved

## Goal

Add internationalization (i18n) support for English and Brazilian Portuguese so the app can be shared with Portuguese-speaking users. Music notation (note names, interval symbols) stays in English as it is universal.

## Library Choice

**i18next + react-i18next + i18next-browser-languagedetector**

Chosen over a homegrown solution because the project will grow and i18next scales well. Provides interpolation, plurals, and language detection out of the box.

## Architecture

### New files

```
src/
  i18n/
    index.ts            # i18next init — detector config, fallback, imports
    locales/
      en.ts             # English strings (source of truth, typed)
      pt-BR.ts          # Brazilian Portuguese strings
```

### Language detection order

1. `localStorage` key `i18nextLng` (user override, persists across sessions)
2. `navigator.language` (`pt-BR` or `pt` → Portuguese; everything else → English)
3. Hardcoded fallback: `en`

### Language switcher

A `PT | EN` toggle added to the bottom of `AppSidebar`. Clicking calls `i18next.changeLanguage()`, which persists to localStorage automatically via the detector plugin.

### Entry point

`src/main.tsx` imports `@/i18n/index` before React renders, so translations are ready synchronously (both locale files are bundled — no async loading needed at this scale).

## Parser Error Refactor

`core/parser.ts` currently returns English error strings. To keep `core/` pure (no language dependencies), errors become structured:

```ts
// src/types/music.ts
export type ErrorCode =
  | "EMPTY_INPUT"
  | "NO_VALID_NOTES"
  | "INVALID_NOTE"
  | "MISSING_ROOT"
  | "INVALID_ROOT_NOTE"
  | "NO_INTERVALS"
  | "INVALID_INTERVAL";

export type ParseError = { code: ErrorCode; token?: string };
```

`ParseOutput` changes from `{ success: false; error: string }` to `{ success: false; error: ParseError }`.

`InputContext` translates the structured error to a localized string using `useTranslation()` before exposing `parseError: string | null` to consumers — no consumer changes needed outside `InputContext`.

## Non-Goals

- Note names (`C`, `D#`, `Bb`) and interval symbols (`b3`, `5`, `maj7`) are NOT translated — they are universal music notation.
- No async locale loading (both locales bundled; app is small).
- No RTL support.
- No locale for anything other than `en` and `pt-BR` at this time.

## Testing

- `TuningControls.test.tsx` must initialize i18next (or mock `useTranslation`) to avoid breaking existing tests.
- No new test cases required beyond keeping existing tests green.
